# 需求澄清策略

## 目标

在进入设计或实现前，把需求收敛到“可执行、可验证、可审查”的状态。

## 核心规则

- 先探索本地事实，再向用户提问。
- 问题必须服务于边界澄清、验收标准、风险取舍或实现决策。
- 模糊需求不允许靠猜测补全。
- 一次优先提出最高影响的 1-3 个问题，避免噪声式追问。

## 必问维度

- 目标：最终要解决什么问题
- 成功标准：如何判断任务完成
- 边界：哪些内容在范围内，哪些明确不做
- 约束：时间、兼容性、性能、安全、流程要求
- 偏好：风格、取舍、默认值、禁止项
- 风险：哪些错误代价最高

## 停止条件

满足以下条件后，才允许结束 `DISCOVERY`：

- `intake_brief` 可完整填写
- 高影响未决项已清零或已明确记录并经用户接受
- 设计阶段不再需要补做需求性决策

## 禁止事项

- 在未完成本地探索前直接把可发现问题抛给用户
- 用单个模糊问题替代系统性澄清
- 在需求不清时进入设计细节或代码实现
