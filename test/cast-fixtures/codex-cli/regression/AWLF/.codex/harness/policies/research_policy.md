# 研究策略

## 目标

让最佳实践、最新信息、推荐和高风险结论建立在可验证来源之上，并把是否采纳的裁决权交给用户。

## 何时强制研究

- 用户要求“最佳实践”“最新信息”“推荐”“对标开源方案”
- 涉及时间敏感事实、规则、版本、产品能力、生态现状
- 涉及高风险领域：安全、法律、财务、医疗、构建链路、公共接口
- 仓库本地证据不足以支撑方案决策

## 研究顺序

1. 本地探索：已有实现、文档、配置、历史约束
2. 外部检索：官方文档、维护者说明、成熟开源实现
3. 候选展开：至少形成 2 个可比较候选，说明其机制、成熟度与适配面
4. 时效校验：记录来源日期和验证日期
5. 留痕：把结论写入 `research_brief` 与 `source_log`

## `research_brief` 必须包含

- 候选方案
- 来源级别
- 维护活跃度
- 风险（若有）
- 机制说明
- 与当前仓库的适配观察
- 需要用户裁决的点

## 明确边界

- 研究负责提供证据和机制解释，不负责替用户做最终采纳裁决。
- 若需要进入实现，采纳结论必须在 `design_packet` 与用户审批阶段显式确认。

## 禁止事项

- 只给一个候选方案却声称完成选型
- 用经验判断替代带来源的事实
- 使用低质量社区帖子覆盖官方文档
