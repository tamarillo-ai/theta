# 架构与系统审查策略

## 目标

强制代理从系统视角设计和审查方案，避免陷入局部最优或补丁式修复。

## 必审视角

- 边界：模块职责是否更清晰，还是更混乱
- 依赖：依赖方向是否正确，是否引入反向耦合
- 数据流：事实源是否唯一，状态是否最小化
- 失败模式：错误传播、降级、回退是否清楚
- 迁移：新接口或新流程如何平滑替换旧行为
- 验证：如何证明方案成立，而不是只证明代码能运行

## 补丁化识别信号

- 通过添加分支、标记位或临时状态规避根因
- 在多个位置重复维护同一份事实
- 新增接口却没有迁移与兼容策略
- 局部修复导致上层/下层语义更混乱
- 设计无法给出可重复验证路径

## 放行条件

- 主方案与备选方案都已比较
- `reuse_decision_matrix` 已完成
- 迁移与回退路径完整
- 系统性风险已被吸收到设计包或明确阻断

## 阻断条件

- 明显的补丁式修复
- 依赖反转错误或边界破坏
- 关键失败模式未覆盖
- 缺失迁移、回退或验证计划
